Guardrails · Semantic Layer

Use the semantic layer. Don't reinvent the joins.

Cube exists so that the way Pelago data joins together is defined once, reviewed, version-controlled, and visible. This page is about why we hold that line, and what to do when you hit something Cube can't answer.

The short version

If the answer can come from Cube, it should.

When you write custom queries against our marts in a BI tool, a notebook, or a spreadsheet, you are recreating join logic that already lives, tested and reviewed, inside Cube. Every time that happens we add a quiet maintenance debt that surfaces later as broken reports, conflicting numbers, and time spent QAing instead of analyzing.

Use Cube views for the join patterns. Bring your own filters, dimensions, and metrics on top. Escalate when something is genuinely missing. That's the contract.

01

Why the semantic layer exists

What you get when joins live in Cube

A semantic layer is a single, governed definition of how Pelago's data fits together. Members join to clients this way. Appointments join to journey markers that way. Once a join is defined in Cube, every report that uses it gets the same answer.

Defined once

One source of truth

The relationship between a member and a client is encoded in Cube. Anyone querying it gets the same answer. No two analysts inventing two slightly different joins.

Code-controlled

Reviewed and versioned

Every change to a Cube view goes through a pull request. We can trace what changed, why, and when, and roll it back if needed.

Validated

Tested before it ships

Schema changes get caught in CI. If an upstream mart changes shape, we know before the dashboards break, not after a stakeholder asks why a number moved.

Visible

Discoverable to everyone

The domain directory shows what Cube can answer. New folks ramp on a documented surface area instead of inheriting someone's untraceable file.

02

The cost of going around it

What happens when queries live outside the semantic layer

A custom query written directly against our marts looks fast. It feels like ownership. The cost shows up later, often after the original author has moved on.

No version control

A query saved inside a BI file, a notebook, or an AI chat can't be reviewed, traced, or rolled back. Nobody outside that surface knows it exists.

Brittle to upstream changes

When a column gets renamed or a mart gets restructured, custom queries break silently. Often the report still runs, but the numbers are wrong.

Inconsistent definitions

The same metric ends up calculated three different ways across three different files. Leadership sees three different numbers and trust erodes.

Hidden dependencies

No one tracks which queries depend on which mart. The data team makes a routine change and a half-dozen unknown reports go red.

Maintenance debt

Whoever inherits the file owns the cleanup. Over-engineered joins that were "quick" a year ago become weeks of untangling now.

QA tax

More time spent verifying numbers than actually analyzing them. The exact thing the semantic layer was built to prevent.

A pattern we want to stop repeating

One overbuilt query inside a BI file is a small problem. Twenty of them is the data team's whole roadmap.

We've already seen what happens when join logic gets baked into a single file with no review and no source control. It can be a BI workbook, a notebook, an AI-generated query, a spreadsheet. The query works at first. The author moves on. The marts evolve. The report ships numbers that look reasonable and aren't, and nobody is sure which fields it was supposed to be joining in the first place.

That's the failure mode this page exists to prevent.

03

Use Cube, or escalate

A simple decision aid

Most analytical questions at Pelago can be answered with Cube. The list below tells you which side you're on.

✓Use Cube when

  • You can express your question as a metric, a dimension, a filter, and a time range.
  • The data lives in a domain you can see in the directory.
  • You need a one-off chart, an answer, or a recurring report.
  • Someone else is likely to ask a similar question.
  • You want the numbers to match what other teams are reporting.

↗Escalate when

  • The field or metric you need doesn't exist in any domain.
  • You need to combine data across domains in a way no view supports.
  • You need to join external data the data team hasn't modeled yet.
  • You're doing exploratory data science that won't be promoted to a recurring report.
  • You think the right answer is a new Cube view.
04

Anti-patterns to avoid

Specific habits that quietly cost us

If you find yourself doing one of these, pause. There's almost always a Cube view that already does it, or a request worth filing instead.

1

Custom SQL written directly against our marts

Anywhere a hand-rolled SQL query touches the warehouse: a BI tool, a notebook, a SQL client, an AI assistant. It doesn't matter where the query was authored. If it bypasses Cube, it bypasses the guardrails.

Why it hurts: the joins are invisible to the rest of the org, the file isn't reviewed, and the query breaks the moment a mart changes.
2

Pulling raw mart tables into a spreadsheet and joining with VLOOKUP

Member table here, client table there, paste into a sheet, VLOOKUP the client name onto the member row.

Why it hurts: manual joins drift the moment one of the source tables updates. There's no audit trail of which version was used.
3

Re-implementing joins inside another tool's modeling layer

Defining the relationship between two tables in a BI tool's data model, in a notebook's join logic, or in a script that becomes a recurring report.

Why it hurts: we end up with two semantic layers. One lives in Cube where it's reviewed and tested. The other lives in someone's file where it isn't. They drift apart. Numbers stop matching.
4

Production reports that depend on a notebook or script outside source control

A Jupyter notebook, a Python script, an ad-hoc query that runs every Monday and emails a number to leadership.

Why it hurts: if the author leaves or the laptop dies, the recurring report dies with it. Nobody can audit how the number is calculated.
5

"One-off" queries that quietly become permanent

The query was for a single board meeting. Now it's the source for a monthly KPI and nobody remembers it was supposed to be temporary.

Why it hurts: permanent reporting belongs in the semantic layer with the same governance as everything else. Drift is guaranteed otherwise.
6

Copy-pasting the same join logic into multiple files

"I already wrote this for the eligibility report, let me paste it into the engagement report too."

Why it hurts: when the join needs to change, only one of the copies gets updated. We've now built a bug into the rest.
05

When Cube can't answer your question

The path that doesn't end in custom SQL

If you've checked the directory and Cube genuinely can't get you the answer, here's what to do instead of writing it yourself.

1

Double-check the domain directory

Skim the cards. Sometimes the field you need lives in a domain you didn't expect (e.g. eligibility data inside Member Journey, not Eligibility).

2

Search the support channel

Someone has probably asked something similar. The thread will either give you the answer or save you re-explaining the use case. While Cube is in UAT, that's #proj-cube-uat. Once we go live, it's #support-data-team.

3

File a request with context

Drop a note in the support channel (#proj-cube-uat during UAT, #support-data-team post-launch) with: what question you're trying to answer, which domains you tried, and what the use case is (one-off vs. recurring report). The data team will route it.

4

If it's recurring, expect a Cube view, not a custom query

The right answer to "I need this every month" is a new view in Cube, not a one-off query you maintain. We'd rather invest the day building it properly than untangle a mart query a year from now.

06

Approved exceptions

When custom queries against marts are okay

There are real cases where mart-level access is the right call.

The point of this page isn't to say "never touch the marts." It's to say "don't promote a custom mart query to production without the same guardrails the semantic layer has." Custom queries are okay when:

  • You're on the data team building or testing a Cube model.
  • You're doing genuine exploratory analysis and the output won't become a recurring report.
  • The query has been reviewed by the data team and lives in a code-controlled location with an owner and an expiry.
  • You've gotten an explicit sign-off because the use case is rare enough that a Cube view isn't justified.

If your custom query doesn't meet one of those, it's an anti-pattern, even if it works today.

Not sure if your use case is covered?

Browse the domain directory or drop a note in #proj-cube-uat (UAT) or #support-data-team (post-launch). We'd much rather build you a clean view than discover a custom query in production six months from now.
