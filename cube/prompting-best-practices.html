Prompting Best Practices for Cube

How to effectively prompt Cube using natural language to get accurate, reliable data outputs. Focused on two critical skills: prompting with specificity, and redirecting prompts when results are incorrect.

This document outlines how to effectively prompt Cube using natural language to get accurate, reliable data outputs. It focuses on two critical skills:

  1. Prompting with specificity
  2. Redirecting prompts when results are incorrect

1Prompting with Specificity

How to structure prompts for accurate queries. This section is not about what to ask, but how to phrase it so Cube correctly translates your intent into the right query.

Core Principle

Cube maps natural language to a data schema. The more your prompt aligns with that schema, the higher the probability of correct results.

Think of prompting as: describing a query in structured language, not asking a question conversationally.

The Prompt Structure Pattern

Use this repeatable structure whenever possible:

Show [metric] by [dimension] over [time range], filtered to [conditions], where [definitions]

This forces clarity across the key components of a query.

Example

Option 1 Show total revenue by month over the last 12 months, filtered to completed orders, where revenue is defined as sum of order_amount
Option 2 Show total_revenue by date_of_service month over the last 12 months, filtered to order_status = 'completed'

Use Schema-Aligned Language

1. Reference Fields Explicitly

Whenever possible, use actual field names and signal that they are fields.

Instead ofrevenue
Prefertotal revenue (sum of order_amount field)
Instead ofregistration date
Preferregistered_date field

This reduces ambiguity and prevents the model from guessing.

2. Be Explicit About Values (Filters)

When filtering, clearly indicate values and where they apply.

Weakonly completed orders
Strongwhere order_status = "completed" (value of order_status field)
WeakUS users
Strongwhere country = "US"

This helps the system distinguish between fields and values.

3. Specify the Metric Calculation

Do not assume the system knows how to compute a metric.

Weakrevenue
Strongtotal revenue defined as sum of order_amount
Weakactive users
Strongactive users defined as count of distinct user_id with at least one login event

4. Define the Time Field and Granularity

Time is one of the most common sources of errors.

  • Specify the field: use order_date as the time field
  • Specify the granularity: grouped by month
  • Specify the range: over the last 12 months

5. Clarify Dimensions Precisely

Avoid vague groupings.

Weakby location
Strongby geographic state (ex value: Missouri)
Weakby customer type
Strongby customer_type field

Further Breakdown

A strong prompt:

  • Maps metric → measure
  • Maps dimension → column
  • Maps filter → field + value
  • Maps time → field + range + granularity

If any of these are ambiguous, the query is likely to be wrong.

Note: It is helpful to familiarize yourself with the difference between different query descriptors (schema elements) such as a filter vs a value vs a field.

When You Don't Know the Field or Value Name

It's common not to know the exact schema. In those cases, guide the model by describing the role of the field and giving example values.

How to Describe an Unknown Field

  • State intent explicitly: I'm looking for a field that represents order status
  • Describe what it does: a field that indicates whether an order is completed, pending, or canceled
  • Provide example values: values like "completed", "pending", "canceled"
  • Ask it to use the closest matching field: use the closest matching field in the schema

How to Describe an Unknown Value

  • Anchor to the field concept: filter to the value that represents completed orders
  • Provide examples or synonyms: values like "complete", "completed", or similar

Combine into a Structured Prompt

Show total revenue by month over the last 12 months, using the field that represents order status (values like "completed", "pending", "canceled"), filtered to the value that represents completed orders
Key Idea

When you don't know exact names, describe what the field represents (its meaning) and what the values look like (examples). This gives the model enough context to map your intent to the correct schema element.

Bad vs Structured Prompt

Conversational (high risk) How is revenue doing recently?
Structured (high confidence) Show total revenue (sum of order_amount) by week over the last 8 weeks, where order_status = "completed"

2Redirecting Prompts When Results Are Incorrect

Even well-structured prompts may return incorrect results due to misinterpretation of terms or schema.

Common Issues

  • Wrong metric selected
  • Incorrect time granularity
  • Missing or incorrect filters
  • Misinterpreted business logic (e.g., "active users")
  • Query join path relationship confusion (most complex issue)

How to Redirect Effectively

A. Clarify the Metric

If the output uses the wrong measure:

Use "total revenue" defined as sum of order_amount, not count of orders

B. Add Missing Constraints

If the scope is wrong:

Limit results to completed orders only and exclude canceled transactions

C. Correct Time Definitions

If time aggregation is incorrect:

Group results by month instead of day

D. Define Business Logic Explicitly

If terms are ambiguous:

Define active users as users who logged in at least once in the past 30 days

E. Iterative Refinement Example

Initial Prompt Show revenue by month
Incorrect Output Issue Includes refunds
Refined Prompt Show monthly total revenue excluding refunded transactions

F. Join Path Refinement

Initial Prompt Show revenue by client industry type
Refined Prompt Show revenue by client industry type. The model seems to be pulling incorrect results: to use the field industry_type you may need to join to the client_profile data set.

3Practical Prompting Checklist

Before submitting a prompt, ask:

  • Did I clearly define the metric?
  • Did I specify time range and granularity?
  • Did I include all necessary filters?
  • Did I remove ambiguity in business terms?
  • Would someone else interpret this the same way?

5Optional Workflow: Use an LLM to Refine the Prompt First

This should not be the default workflow for every prompt. Most of the time, the goal is to write one clear prompt and get the answer back.

However, if you are struggling to create a strong prompt, you can use an LLM to refine or optimize the prompt before running it in Cube. This can also be a useful practice technique. By asking an LLM to improve your prompt, you can learn how to phrase analytics questions with more structure, specificity, and schema alignment.

Why This Works

LLMs are often strong prompt engineers. They can help translate a rough question into a more structured prompt with clearer metrics, dimensions, filters, field names, values, and definitions.

As workflows move toward more agent orchestration, allowing one model to optimize the prompt before another model executes against data can improve consistency and reliability.

When to Use This Workflow

Use prompt refinement when:

  • You are struggling to phrase the prompt clearly
  • The first result is wrong or ambiguous
  • The question involves multiple filters, metrics, or time windows
  • You want to practice writing more structured analytics prompts
  • The output needs to be especially reliable

6Summary

High-quality prompts are
  • Specific
  • Structured
  • Explicit about fields, values, definitions, logic, and constraints

When results are incorrect, treat prompting as an iterative process. Refine inputs by clarifying assumptions, adding constraints, and correcting definitions.

The more closely your prompt maps to the underlying data schema, the more reliable your output.
